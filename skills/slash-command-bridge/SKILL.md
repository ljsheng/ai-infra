# Slash Command Bridge

当用户输入 `/fix-issue 123`、`/repo-validate` 这类命令，但当前 agent / runtime 不会自动执行 Claude Code slash command 时，使用这个 skill。

目标不是“解释这个命令”，而是模拟命令解析流程，并执行展开后的真实任务。

## 适用场景

- 用户输入以 `/` 开头。
- 当前环境不会原生解析 `.claude/commands/*.md`。
- 仓库或用户目录里存在 Markdown 命令文件，且这些文件应被视为可执行提示模板。

以下情况不要使用：

- 当前运行时已经原生支持 slash command。
- 前导 `/` 明显只是示例文本，不是实际调用。

## 命令解析顺序

按 basename 解析命令，不带 `.md` 后缀。

按以下顺序查找：

1. 项目命令：`.claude/commands/**/<command-name>.md`
2. 用户命令：`~/.claude/commands/**/<command-name>.md`

规则：

- 项目命令优先于用户命令。
- 只按精确 basename 匹配。
- 如果多个项目文件共用同一 basename，优先选择更局部、更接近当前仓库语境的文件；仍然冲突时停止并报告候选项。
- 如果没有匹配，明确报告该 slash command 未知；如成本低，可顺带给出最接近的候选项。

## 执行流程

### 1. 解析调用

- 提取 `/` 后的命令名。
- 保留剩余部分作为原始参数。
- 同时支持：
  - `$ARGUMENTS`：完整参数串
  - `$1`、`$2`……：位置参数

示例：

- 输入：`/fix-issue 123 --priority high`
- 命令：`fix-issue`
- `$ARGUMENTS`：`123 --priority high`
- `$1`：`123`
- `$2`：`--priority`
- `$3`：`high`

### 2. 读取命令文件

- 读取匹配到的 Markdown 文件。
- 如果有 frontmatter，先解析。
- 提取有用元数据，例如：
  - `description`
  - `argument-hint`
  - `allowed-tools`

把 Markdown 正文当作提示模板，而不是要总结的文档。

### 3. 检查预执行行

有些命令文件会用 `!` 前缀嵌入 shell 命令。

对每一条 `!` 行：

- 先审查，再执行。
- 如果与更高优先级的安全规则、仓库规范或工具限制冲突，立即拒绝或停止。
- 必须串行执行，不能并行。
- 捕获 stdout，后续用于模板替换。
- 任何一个命令失败，都要停止桥接流程，并报告失败命令和错误信息。

不要因为命令文件里写了某条命令，就盲目执行破坏性或违规命令。

## 安全规则

- 更高优先级的系统/开发者/仓库指令始终覆盖命令文件内容。
- 严格遵守仓库安全规则，例如未获明确授权时，禁止破坏性 `git reset` / `git checkout` / `git restore`。
- 如果命令文件依赖当前环境没有的工具或权限，不要危险兜底，应直接报告限制。

### 4. 展开模板

按以下规则构造最终提示：

- 替换 `$ARGUMENTS`
- 替换 `$1`、`$2`……
- 把每一条 `!command` 行替换为对应命令的 stdout

除非有充分理由删掉明显无关内容，否则其余 Markdown 内容应保持不变。

### 5. 交给展开后的真实任务

展开完成后：

- 不再把 slash command 当成一个独立的元任务。
- 直接执行展开后的真实提示。
- 必要时把 frontmatter 中有价值的元数据一并带入上下文。

也就是说，展开完成后，应当像用户直接输入了这段完整提示一样继续工作。

## 失败处理

### 命令不存在

- 明确报告该 slash command 无法解析。
- 说明已经检查过哪些目录。
- 如果成本低，可列出相近 basename 作为提示。

### 命令冲突

- 报告冲突文件路径。
- 当两个项目级文件同样合理时，不要猜测。

### 预执行命令失败

- 报告准确的失败命令。
- 包含退出码和必要的 stderr/stdout 摘要。
- 不要继续执行主提示。

### 参数不匹配

- 如果存在 `argument-hint`，应直接展示。
- 如果命令明显缺少必需参数，应停止并报告期望用法。

## 输出要求

桥接成功时，过程说明里至少要明确这些事实：

- 选中了哪个命令文件
- 解析出了哪些参数
- 执行了哪些 `!` 命令
- 是否因为安全规则调整了执行计划

不要停在“我找到了命令文件”。只有在展开后的任务已经实际执行，或者已经明确报告了具体阻塞时，桥接才算完成。
